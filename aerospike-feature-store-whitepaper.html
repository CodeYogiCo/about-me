1.Introduction: What Is a Feature Store and Why It Matters

A feature store is a centralized data system purpose-built to manage the lifecycle of machine learning features — from offline computation to online serving. It bridges the gap between batch data pipelines and real-time inference, providing a single source of truth for feature values consumed by both training and serving paths.

At its core, a feature store must satisfy two competing contracts simultaneously:

ContractRequirement
Offline storeHigh-throughput historical reads for model training and backfills
Online storeSub-millisecond point lookups at high QPS under strict p99 SLAs

The tension is fundamental. Training jobs need columnar scans over billions of rows; serving systems need single-key reads at millions of requests per second. Most systems resolve this by running two separate stores — a columnar warehouse (e.g., BigQuery, Hive, Parquet on S3) for the offline path and a low-latency KV store for the online path, with a materialization pipeline syncing between them.

This white paper focuses exclusively on the online feature store tier and how we use Aerospike to serve real-time features into search ranking.

2.Feature Stores in the Search Ranking Stack

2.1 The Ranking Pipeline at a Glance

Search ranking is a multi-stage retrieval and scoring pipeline:

Query │ ▼ [Recall / ANN Retrieval] ← sparse (BM25) + dense (HNSW / ScaNN) │ ▼ [L1 Lightweight Ranker] ← fast, low-dimensional features │ ▼ [L2 Full Ranker] ← deep ML model (GBDT, DNN, or LTR) │ ← feature store read happens here ▼ [Post-Ranker / Business Logic] │ ▼ Results

At L2 — the full ranker — we consume hundreds of features per (query, document) pair. These features fall into three categories:

2.1.1 Query-side Features

Real-time signals derived from the query and session context:

  • Query intent embedding (dense vector from query encoder)
  • Query historical CTR, query popularity percentile
  • User's session context: recent clicks, dwell time, zero-result history
  • User's long-term behavioral segments (categorical + continuous)

2.1.2 Document-side Features

Pre-computed signals about each candidate document:

  • Document quality score (freshness-adjusted PageRank equivalent)
  • Engagement statistics: CTR@k, NDCG contribution, dwell time distribution
  • Content signals: keyword density, entity coverage, recency
  • Business signals: seller trust score, inventory status, revenue tier

2.1.3 Cross Features

Interaction signals between query and document:

  • Historical (query, document) co-occurrence CTR
  • Semantic similarity bin (bucketed cosine distance)
  • Category affinity score

2.2 Latency Budget Constraints

A production search system's end-to-end SLA is typically 200–300ms. The L2 ranker's allocated budget is 30–50ms total, of which feature retrieval must consume no more than 5–10ms at p99. This is non-negotiable — every millisecond of budget overage either degrades user experience or forces retrieval cutoffs that hurt recall.

At a typical L2 cutoff of 200–500 candidates, we issue:

  • 1 query-side lookup (keyed by user_id or session_id)
  • 200–500 document-side lookups (keyed by doc_id)
  • Up to N cross-feature lookups (keyed by (query_hash, doc_id))

That is 200–1000 individual key lookups per search request, at tens of thousands of QPS globally. The online feature store must sustain:

MetricRequirement
Throughput1M+ reads/sec sustained, 10M+ peak
Latencyp50 < 1ms, p99 < 5ms, p999 < 10ms
Availability99.99%+
Data sizeTerabytes (not DRAM-friendly at scale)

2.3 Feature Freshness Requirements

Not all features carry the same freshness tolerance:

Feature ClassFreshness RequirementUpdate Frequency
Real-time engagement (CTR last 1h)< 5 min staleStreaming (Kafka → Flink)
Session context< 30s staleEvent-driven write-through
Daily aggregates< 24h staleBatch (Spark job)
Static document signalsUp to 7 days tolerancePeriodic recompute

The feature store must handle both streaming micro-batch writes and bulk batch upserts without impacting read latency.

3.Why Aerospike

3.1 The Problem With Naive Solutions

Before committing to Aerospike, we evaluated the conventional alternatives:

Redis (standalone or Redis Cluster): Redis is the default go-to for low-latency KV. It delivers excellent throughput for DRAM-resident data. However, at our data volumes — terabytes of pre-computed features — the economics break down fast. All data must live in DRAM. At $3–5/GB/month for memory-optimized EC2 instances, storing 5TB of feature data runs into millions of dollars annually just for the memory tier. Horizontal sharding across hundreds of Redis shards introduces operational complexity and cross-shard coordination overhead.

DynamoDB / Cassandra: Suitable for large data volumes but latency characteristics are wrong for our SLA. p99 latencies on DynamoDB regularly hit 10–20ms under real traffic. Cassandra is better-tuned but still oriented toward durability over microsecond retrieval.

In-process cache: Each ranking service instance maintains a local LRU cache. This works for hot documents but thrashes on the long tail. In search ranking, the bottom 80% of documents account for most traffic by query diversity.

3.2 Aerospike's Value Proposition

Aerospike's design philosophy solves exactly the tradeoff we face: DRAM-speed reads for hot data, SSD-persisted capacity for the long tail, at a fraction of the cost of pure DRAM solutions.

Three architectural decisions make Aerospike uniquely suited for this use case:

  1. Hybrid Memory Architecture (HMA) — indexes live in DRAM; values live on NVMe SSDs. Read paths are single I/O operations.
  2. Native multi-row batch reads — a single RPC can fetch hundreds of keys with minimal per-key overhead.
  3. Strong consistency with configurable replication — feature writes from Flink pipelines are durably committed, not fire-and-forget.

4.Aerospike Architecture Deep Dive

4.1 Storage Architecture: Hybrid Memory Architecture (HMA)

The foundational innovation in Aerospike is the separation of the primary index from record data:

┌─────────────────────────────────────────────────────────┐ │ DRAM │ │ │ │ Primary Index (in-memory red-black tree per partition) │ │ ┌─────────────────────────────────────────────────┐ │ │ │ key hash → { device_id, file_id, offset, size }│ │ │ └─────────────────────────────────────────────────┘ │ │ │ │ Row Cache (optional, LRU, configurable size) │ └─────────────────────────────────────────────────────────┘ │ │ single direct I/O ▼ ┌─────────────────────────────────────────────────────────┐ │ NVMe SSD Layer │ │ │ │ Aerospike Storage Engine (log-structured) │ │ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ │ │ Partition │ │ Partition │ │ Partition │ ... │ │ │ File 0 │ │ File 1 │ │ File 2 │ │ │ └────────────┘ └────────────┘ └────────────┘ │ └─────────────────────────────────────────────────────────┘

Why this matters for us:

The primary index for our entire document feature dataset (billions of documents) fits in DRAM because it only stores the pointer to the record, not the record itself. A typical index entry is ~64 bytes. At 10B documents, that is ~640GB of DRAM for the index — distributed across a cluster, ~20–40 nodes at 32GB DRAM each. The multi-terabyte feature payload lives on NVMe, which at $0.10/GB/month is 30–50× cheaper than DRAM.

A read operation proceeds as follows:

  1. Hash the key → locate the index entry in DRAM (nanoseconds)
  2. Resolve device offset from index entry
  3. Issue a single pread() direct I/O call to NVMe (50–150µs on modern NVMe)
Key insight: Direct I/O bypasses the OS page cache (no double buffering). Aerospike pre-allocates storage in large contiguous extents to minimize fragmentation and seek overhead. This is what makes sub-millisecond reads from SSD achievable in practice.

4.2 Data Model

Aerospike uses a namespace → set → record hierarchy analogous to database → table → row:

Namespace: "feature_store"
  ├── Set: "user_features"
  │     Key: user_id (string or integer digest)
  │     Bins: { "ctr_1h": 0.031, "session_depth": 4, "query_vec": [0.1, 0.4, ...] }
  │
  ├── Set: "doc_features"
  │     Key: doc_id
  │     Bins: { "quality_score": 0.87, "ctr_7d": 0.024, "freshness": 0.95 }
  │
  └── Set: "cross_features"
        Key: sha256(query_hash + doc_id)
        Bins: { "hist_ctr": 0.015, "sem_sim_bin": 3 }

Bins are strongly typed: integer, double, string, blob, list, map, CDT (Collection Data Types). This means we can store complex feature values like embedding vectors (as blob or list) without extra serialization overhead on the client.

4.3 Batch Read Path

The killer feature for our use case is native batch reads. A single batch read RPC to Aerospike can contain hundreds of keys, routed to their respective cluster nodes in parallel: 

// Aerospike Python client — batch read
keys = [
    ("feature_store", "doc_features", doc_id)
    for doc_id in candidate_doc_ids   # 200–500 doc IDs
]

records = client.get_many(keys)
# Returns list of (key, metadata, bins) tuples
# Single round-trip per node in the cluster

Internally, the client partitions keys by which cluster node owns them (consistent hashing via RIPEMD-160 digest), sends one request per node in parallel, and reassembles the responses. For a 10-node cluster with uniform distribution, a 500-key batch requires 10 parallel RPCs instead of 500 sequential ones. This reduces round-trip amplification by 50×.

OperationMedianp99
500-key batch read2.1ms6.8ms
100-key batch read0.8ms2.9ms
Single-key read0.3ms1.1ms

Measured over 10GbE within the same AZ on a 10-node production cluster.

4.4 Cluster Architecture and Replication

Aerospike uses a shared-nothing, masterless architecture based on a Paxos-derived clustering protocol:

┌─────────┐ ┌─────────┐ ┌─────────┐ │ Node 1 │────│ Node 2 │────│ Node 3 │ │ (AS-1) │ │ (AS-2) │ │ (AS-3) │ └─────────┘ └─────────┘ └─────────┘ │ │ │ Partition Partition Partition [0..1023] [1024..2047] [2048..3071] + replicas + replicas + replicas

Each namespace is divided into 4096 partitions. With replication factor 2, each partition has a master and one replica. Writes go to the master and either synchronously replicate to the replica before acknowledging (COMMIT_ALL — strong consistency) or asynchronously (COMMIT_MASTER — lower write latency).

For feature store writes from our Flink streaming pipeline:

  • Replication factor = 2 cluster-wide
  • COMMIT_ALL for session features (strong consistency required)
  • COMMIT_MASTER for bulk batch-loaded document features (throughput over strict consistency)

4.5 Write Paths: Streaming vs. Batch Ingestion

Streaming path (Flink → Aerospike)

Real-time features are emitted by Flink jobs consuming from Kafka. Each Flink task manager runs an Aerospike client and issues individual or small-batch writes: 

Kafka Topic: user_engagement_events
    │
    ▼
Flink: aggregate CTR per (user_id, query_category)
       over 1h tumbling window
    │
    ▼
Aerospike Sink: upsert to "user_features" set
    policy: upsertOnly, TTL=86400s, generation check disabled

Write throughput from our Flink fleet: sustained 80K writes/sec, peak 300K/sec during traffic spikes. Aerospike handles this without backpressure.

Batch path (Spark → Aerospike)

Daily document feature recomputation runs as a Spark job that reads from Hive/Parquet and bulk-loads into Aerospike via the Aerospike Spark Connector: 

df.write \
  .format("aerospike") \
  .option("aerospike.seedhost", "aerospike-cluster:3000") \
  .option("aerospike.namespace", "feature_store") \
  .option("aerospike.set", "doc_features") \
  .option("aerospike.updateByKey", "doc_id") \
  .save()

# Throughput: ~1.2M records/sec across a 20-node cluster

4.6 Expiry and TTL Management

Feature freshness is enforced at the storage level via per-record TTL. Aerospike's background vacuum thread evicts expired records without impacting read latency:

Feature TypeTTL SettingRationale
Session features3600s (1h)Auto-evicted when stale
Real-time engagement300s (5m)Short freshness window
Document features (batch)0 (no expiry)Managed by batch refresh cycle

This removes the need for a separate cache invalidation system.

4.7 Cross-Datacenter Replication (XDR)

Aerospike's built-in Cross-Datacenter Replication (XDR) enables asynchronous replication between geographic regions:

  • Primary cluster in us-east-1 (write source)
  • Replica clusters in eu-west-1 and ap-southeast-1 (read-optimized)
  • XDR lag: typically 50–200ms — within our feature freshness SLAs

Ranking services in each region read from their local Aerospike cluster, keeping feature reads within the same AZ and eliminating cross-region read latency.

5.2 When Redis Is Still the Right Choice

  • Dataset fits comfortably in DRAM (< 50GB) — Redis DRAM-native reads (0.1ms) are faster and operationally simpler.
  • Rich data structure operations are required — Sorted sets (ZADD/ZRANGE), HyperLogLog, Pub/Sub — no Aerospike equivalent.
  • Absolute minimum latency (p99 < 1ms) — For use cases requiring sub-millisecond p99 on every single read, Redis DRAM wins.
  • Team has deep Redis expertise — Operational familiarity has real cost; don't switch without accounting for it.

5.3 When Aerospike Is the Right Choice

  • Data volume exceeds DRAM economics — Above 100GB, Aerospike's hybrid storage makes it economically viable. Above 1TB, there is no contest.
  • Batch reads across large candidate sets — Ranking 500 documents per query with a single batch RPC is Aerospike's native pattern.
  • Strong cross-datacenter replication without a paid tier — Aerospike XDR is open-source. Redis active-active geo-distribution requires Redis Enterprise.
  • Streaming writes and batch bulk loads coexist — Aerospike handles both concurrently. Redis requires operational separation (write-through cache + separate persistence layer).
  • Long-tail data access patterns — In search ranking, most documents are low-frequency. DRAM caches thrash on the long tail. Aerospike's NVMe tier handles arbitrary access patterns at low latency.
  • Operational scalability — Aerospike rebalances automatically when nodes are added. Redis Cluster resharding is a manual operational event with risk.

5.4 Our Decision Rationale

Our document feature dataset at the time of migration was 2.8TB and growing at 400GB/quarter. A Redis cluster to hold this would require approximately 280 r6g.2xlarge nodes (64GB DRAM each, with headroom). At $0.504/hr per node, that is $1.1M/year in compute alone.

Our Aerospike cluster serving the same data: 24 nodes with 2TB NVMe each, 32GB DRAM per node for index. Total NVMe: 48TB (headroom for 2× growth). Cost: ~$180K/year.

Result: 83% infrastructure cost reduction. Batch read p99 improved from 12ms (Redis pipelining across shards) to 6.8ms (Aerospike native batch), because we eliminated the client-side shard coordination layer entirely.

7.Summary and Recommendations

Aerospike as an online feature store in the search ranking stack delivers the rare combination of low-latency reads, large data capacity, and operational scalability that no single-tier solution achieves.

The key architectural insight is that ranking workloads have two properties Aerospike is purpose-built for: high fan-out batch reads (hundreds of document lookups per query) and large long-tail datasets (billions of documents, multi-TB feature space) that cannot be economically held in DRAM.

Adopt Aerospike for your online feature store when:
  • Your feature dataset exceeds 100GB and is growing
  • You rank > 100 candidates per query with per-document feature lookups
  • You run both streaming and batch feature ingestion pipelines
  • You operate across multiple geographic regions
  • p99 feature read latency must stay under 10ms
Stick with Redis when:
  • Dataset fits in DRAM with room to spare
  • You need Redis-native data structures (sorted sets, pub/sub)
  • Absolute minimum single-key latency (< 0.5ms p99) is required
  • Team investment in Redis operations is already high

The migration from Redis Cluster to Aerospike in our ranking stack reduced infrastructure cost by 83%, improved batch read p99 by 43%, and eliminated the operational burden of manual Redis resharding as our feature corpus grows.
